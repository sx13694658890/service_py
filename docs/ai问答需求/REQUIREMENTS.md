# AI 问答（首页聊天窗口）需求说明

本文档描述在 **Web 首页** 增加的 **AI 问答聊天窗口** 的产品目标、范围与验收标准。技术契约若涉及后端接口，需与 OpenAPI 及 [FRONTEND.md](../FRONTEND.md) 中的通用约定（认证、错误格式等）保持一致或单独成册。

---

## 1. 背景与目标

| 项 | 说明 |
|----|------|
| 背景 | 用户在首页需要快速获得产品/使用说明类帮助，降低跳转成本 |
| 目标 | 提供可发现、可关闭、可继续会话的 **AI 问答** 能力，默认在首页可用 |
| 成功标准 | 用户可在 3 步内完成「打开窗口 → 提问 → 看到回答」；关键路径无明显卡顿与错误吞没 |

---

## 2. 范围与非目标

### 2.1 本期范围（MVP）

- 首页展示 **聊天入口**（浮层按钮，可固定在视口一角，默认可配置为左下或右下）。
- 点击入口后展开 **聊天面板**（弹层/抽屉式，不整页跳转）。
- **消息列表**：区分用户消息与 AI（助手）消息；支持多轮上下文（同一会话内）。
- **文本输入**与 **发送**；发送后展示加载/进行中状态。
- **快捷问题**（可选配置）：展示若干预设问题 chip，点击即作为用户消息发送或填入输入框（产品定一种默认行为，文档以「直接发送」为推荐）。
- **关闭**面板；再次打开时策略：**默认保留本会话消息**（刷新页面后是否保留见 §6.2）。

### 2.2 非目标（可列后续迭代）

- 富文本编辑器、@ 提及、多人协作房间。
- 语音输入/输出、视频消息。
- 完整「人工客服排队与转接」系统（若仅需占位按钮，可作为 UI 壳 + 提示文案）。
- 表情包选择器、大文件附件上传（可作为后续；MVP 可隐藏或禁用）。
- 保证大模型回答 100% 事实正确（需通过提示词、引用来源、免责声明等方式管理预期）。

---

## 3. 用户与场景

### 3.1 主要角色

- **访客**：未登录用户，仅可使用公开额度内的问答（若后端限流/限次）。
- **已登录用户**：可携带身份调用后端，享受更高配额或个性化（若产品需要）。

### 3.2 典型场景

1. 用户浏览首页，看到浮标，点击打开聊天窗，选择快捷问题「这是什么？有什么用？」获得说明。
2. 用户在输入框输入自定义问题，发送后等待流式或整段回复，再继续追问。
3. 用户关闭窗口后继续浏览；再次打开仍能看到刚才的对话记录（同标签页内）。

---

## 4. 功能需求

### 4.1 入口（Floating Trigger）

- **形态**：圆形主按钮 + 对话气泡图标；有轻微阴影，与首页背景（含 `banner`）对比度足够。
- **位置**：默认 **固定于视口** 一角（建议可配置 `bottom-left` / `bottom-right`，避免遮挡主 CTA）。
- **可访问性**：支持键盘聚焦；`aria-label` 如「打开 AI 助手」。
- **状态**：打开面板时入口可变为「收起」或隐藏（二选一，产品定稿；推荐保留最小化态以便再次展开）。

### 4.2 聊天面板（Panel）

- **布局**（对齐常见客服/助手 widget）：
  - **顶栏**：标题（如「AI 助手」）、在线状态指示（可选）、**关闭**按钮。
  - **消息区**：可滚动列表；用户消息与助手消息 **视觉区分**（对齐方式、头像/图标可选）。
  - **快捷问题区**（可选）：横向换行或纵向列表的 **chip 按钮**。
  - **底栏**：多行文本输入框、**发送**按钮；附件/表情若不做则隐藏。
- **尺寸**：桌面端建议宽度约 360–420px、高度不超过视口约 70%；小屏全屏或底部抽屉需响应式说明（验收中列出断点）。
- **层级**：`z-index` 高于首页常规内容，低于全局 Modal/全屏错误页（避免死锁）。

### 4.3 对话与 AI 能力

- **请求**：将用户消息（及可选上下文消息列表）提交后端 **AI 问答接口**。
- **响应**：
  - **MVP** 可支持 **一次性返回** 完整文本；
  - **增强** 支持 **SSE/流式** 输出（逐字/逐段显示，需单独排期）。
- **异常**：网络失败、429、5xx、业务错误需有 **可理解提示**；错误格式建议沿用 FastAPI `detail` 约定（见 [FRONTEND.md](../FRONTEND.md) §5）。
- **空状态**：无消息时的欢迎语 + 快捷问题引导。

### 4.4 内容与合规

- **免责声明**：面板内或首次展开时简短说明「AI 生成内容仅供参考」。
- **敏感内容**：依赖后端策略与（可选）前端关键词拦截；具体规则由安全/法务与后端共同定义。
- **日志与隐私**：是否记录对话、保留时长、脱敏要求需在隐私政策或技术方案中写明（本文档仅提需求项）。

### 4.5 与现有工程的关系

- **首页集成点**：聊天组件挂载于首页布局内（如 `HomePage`），**不强制**登录即可打开（若产品要求必须登录，则在需求中改为「未登录引导去登录」）。
- **设计系统**：与现有 **React + antd + Tailwind** 一致；复杂交互可封装到 `packages/ui` 或 `apps/web` 下独立模块。

## 5. 验收标准（MVP Checklist）

- [ ] 首页可见浮标，点击后面板打开/关闭符合 §4.1–4.2。
- [ ] 用户可发送文本，助手侧返回可见结果（对接真实或 Mock 接口均可先验收 UI，最终以联调为准）。
- [ ] 快捷问题可点击并触发至少一条助手回复（真实或 Mock）。
- [ ] 错误场景有明确提示，不出现白屏或无反馈一直转圈。
- [ ] 基础可访问性：焦点可见、关闭可操作、主要区域可用键盘到达（输入框发送若需快捷键可作为加分项）。
- [ ] 与 [FRONTEND.md](../FRONTEND.md) 冲突时，以 **OpenAPI 契约为准** 并更新本文档。

## 6. 待定事项（需产品/后端确认）

| 编号 | 议题 | 选项/建议 |
|------|------|-----------|
| T1 | 是否必须登录 | MVP 常做「可选登录」：未登录匿名会话 + 限额 |
| T2 | 会话持久化 | 仅 `sessionStorage` / `localStorage` / 完全内存；是否上送 `conversation_id` |
| T3 | 流式输出 | MVP 整段 vs 一期即上 SSE |
| T4 | 模型与后端 | 自建代理 / 第三方 API；密钥仅存服务端 |
| T5 | 快捷问题文案与配置 | 前端写死 vs CMS/后端下发 |
| T6 | 浮标默认位置 | 左下 / 右下；是否跟随 RTL |
