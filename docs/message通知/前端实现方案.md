# 站内消息通知 — 前端实现方案

本文档依据已落地的服务端能力（见同目录 [IMPLEMENTATION_PLAN.md](./IMPLEMENTATION_PLAN.md)），描述**前端**如何对接与实现产品形态。  
HTTP 形态、鉴权与路径速查见仓库根目录 [FRONTEND_API.md](../FRONTEND_API.md)；工程约定见 [FRONTEND.md](../FRONTEND.md)。

---

## 1. 目标与范围

### 1.1 目标

- 登录用户在顶栏（或等价入口）看到**未读角标**，可打开通知列表。
- 支持**拉取列表**、**单条已读**、**全部已读**、**软删除**（`204`）。
- 可选：建立 **SSE 长连接**，在新通知或未读数变化时**即时刷新 UI**，减少轮询。
- 与现有 **Bearer JWT**、Axios（或 fetch）封装、401 统一处理保持一致。

### 1.2 本期不做（边界）

- 不实现短信、邮件、浏览器 Push。
- 不做富文本模板引擎；正文按纯文本展示即可（后续可扩展 Markdown）。
- 服务端已配置 **`REDIS_URL`** 时，通知经 **Redis Pub/Sub** 广播，多 Worker 下 SSE 仍可收到推送；未配置 Redis 时仅**单进程**内 SSE 实时，前端应保留**定时拉未读数**兜底（见第 5 节）。

## 2. 与服务端契约对齐

### 2.1 鉴权

- 所有 `/messages` 相关请求需头：`Authorization: Bearer <access_token>`。
- `401`：清本地会话并跳转登录（与现有全局拦截器一致）。

### 2.2 REST 接口（相对 `{API_V1_PREFIX}`，默认 `/api/v1`）

| 场景 | 方法 | 路径 | 说明 |
|------|------|------|------|
| 列表 | GET | `/messages` | Query：`limit`（默认 20，最大 100）、`offset`、`only_unread` |
| 未读数 | GET | `/messages/unread-count` | `{ "unread_count": number }` |
| 单条已读 | POST | `/messages/{message_id}/read` | 幂等；`{ "message": "ok" }` |
| 全部已读 | POST | `/messages/read-all` | `{ "updated": number }` |
| 删除 | DELETE | `/messages/{message_id}` | **204**，无 body |

### 2.3 列表项字段（`items[]`）

- `id`：字符串 UUID，即路径中的 `message_id`。
- `category`：如 `security` / `system` / `business`（展示可做图标或色条映射）。
- `title`、`content`、`payload`（可选 JSON）、`priority`（`low` | `normal` | `high`）。
- `is_read`、`read_at`（ISO 时间或 null）、`created_at`（ISO 时间）。

响应同时包含：`total`（与当前筛选一致的总条数）、`unread_count`（当前用户未读总数，与是否 `only_unread` 无关）。

### 2.4 SSE：`GET /messages/stream`

- **GET**，可携带与 REST 相同的 Bearer；浏览器可使用原生 **`EventSource`**（注意：仅 GET，与 AI 的 POST SSE 不同）。
- 媒体类型：`text/event-stream`；每行 `data: ` 后为 **单行 JSON**（与现有 AI SSE 解析习惯一致）。
- JSON 根字段 `type`：
  - **`notification`**：`item` 为一条与列表项结构一致的通知快照（用于增量插入或刷新列表）。
  - **`unread_count`**：`unread_count` 数字，用于更新角标。
  - **`heartbeat`**：保活，可忽略或用于连接状态指示。

**注意**：未启用 Redis 时，多实例部署下 SSE 仅覆盖连接所在进程；启用 Redis 后由 `msg:notify:{userId}` 频道统一分发。仍建议**低频轮询** `unread-count` 或在「打开抽屉」时强制 `GET /messages` 作兜底。

## 3. 推荐前端架构

### 3.1 分层

1. **API 层**（如 `packages/api`）：`messages.api.ts`，封装上述 REST +（可选）SSE URL 构建。
2. **状态层**：未读数、列表、连接状态；可用 React Context、Zustand、或 TanStack Query（推荐 Query 管理缓存与重试）。
3. **UI 层**：顶栏 `Badge` + `Popover`/`Drawer` + `List`，操作按钮触发对应 API。

### 3.2 TypeScript 类型（建议）

与 OpenAPI 对齐即可；最小手写示例：

```ts
export type MessagePriority = 'low' | 'normal' | 'high';

export interface MessageItem {
  id: string;
  category: string;
  title: string;
  content: string;
  payload?: Record<string, unknown> | null;
  priority: MessagePriority;
  is_read: boolean;
  read_at: string | null;
  created_at: string;
}

export interface MessageListResponse {
  items: MessageItem[];
  total: number;
  unread_count: number;
}
```

## 4. 交互与 UI 建议

### 4.1 顶栏入口

- 铃铛图标 + `Badge` 显示 `unread_count`（0 时可隐藏数字或显示点）。
- 点击打开 **Popover**（少量）或 **Drawer**（分页/滚动）：与产品设计一致即可。

### 4.2 列表行为

- 默认按服务端顺序展示（服务端按投递时间倒序）。
- 点击一条：
  - 若 `!is_read`，调用 `POST .../read`，成功后本地置 `is_read: true` 并减少未读数（或重新拉 `unread-count`）。
- 「全部已读」：`POST /read-all`，成功后刷新列表或本地批量更新。
- 删除：`DELETE`，成功后从列表移除；未读数若变化可再拉 `unread-count`。
- **空态**：无数据时展示引导文案；**失败态**：`message.error` + 可重试。

### 4.3 `payload` 与跳转（可选）

- 例如改密通知：`payload.kind === 'password_changed'` 可显示「账户安全」标签或跳转 `/dashboard/account`。
- 跳转逻辑应**防御性**：未知 `kind` 忽略，避免硬编码崩溃。

### 4.4 `priority` 与 `category` 映射

- `high`：列表项左侧色条或标题加粗。
- `category === 'security'`：安全类图标（与改密等事件一致）。

## 5. 数据刷新策略（轮询 + SSE）

### 5.1 最小可用（仅轮询）

- 进入已登录布局后：定时 `GET /messages/unread-count`（如 45s）。
- 打开抽屉时：`GET /messages` 拉第一页。

### 5.2 推荐（SSE + 兜底）

1. 用户登录且进入 Dashboard 后建立 `EventSource`（URL 需带同源代理下的 `/api/v1/messages/stream`，**Token 传递见下节**）。
2. `onmessage`：解析 JSON，按 `type` 更新全局 `unread_count` 或往列表头部合并 `notification.item`（注意去重：`id`）。
3. `onerror`：关闭并重连（指数退避，上限如 30s），同时**不关闭**定时轮询兜底。
4. 登出时：关闭 `EventSource`，清空本地通知状态。

### 5.3 SSE 携带 JWT 的两种做法

- **Cookie 会话**：若未来改为 HttpOnly Cookie，EventSource 同域即可（当前若仍为 Bearer，需二选一）。
- **Query Token（仅在内网或短期）**：部分网关允许 `?access_token=`；**注意安全**，优先由**反向代理**注入身份或由后端提供**短期 SSE 专用票据**（当前服务端为 Header Bearer，若原生 EventSource 无法加 Header，需通过 **fetch + ReadableStream** 解析 SSE，与 AI 流式类似，或让网关把 Authorization 转发规则与文档对齐）。

**落地建议**：若团队坚持 Header Bearer，对 `/messages/stream` 使用 **`fetch` + stream 解析**（复用 AI SSE 工具函数），避免把长期 Token 写在 URL。

## 6. 与业务场景联调

- **修改密码成功**：服务端会投递 `category: security` 通知；用户若在其它页停留，依赖 SSE 或下一轮询更新角标。
- **多标签页**（可选进阶）：`BroadcastChannel` 或 `storage` 事件同步未读数，避免重复请求；非必须。

---

## 7. 测试清单（前端）

- 未登录访问 `/messages` → 401 → 跳转登录。
- 列表分页：`limit`/`offset` 与 `total` 一致；`only_unread=true` 时仅未读。
- 单条已读幂等：连续点击不报错，未读数正确。
- 删除后 204，列表与未读一致。
- SSE：模拟收到 `unread_count` 与 `notification`，UI 更新；断网重连后兜底轮询仍可用。
