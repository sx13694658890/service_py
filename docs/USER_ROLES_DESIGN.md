# 用户角色设计思路

本文档描述在现有「邮箱 + 密码 + JWT」体系上，为用户增加**角色（Role）**的推荐建模与演进方式，供实现与评审使用。不涉及具体代码实现细节，落地时以 Alembic 迁移与接口契约为准。

**前端联调**：HTTP 形态、JWT 头与载荷字段说明见 [FRONTEND.md](./FRONTEND.md)。

---

## 1. 目标与边界

### 1.1 目标

- 用户可拥有**一个或多个**角色（多对多）。
- 角色用于**鉴权**：部分接口仅允许特定角色访问。
- 与现有 `users` 表、登录/注册流程兼容，迁移可增量上线。

### 1.2 边界（本期不展开）

- **权限细粒度到「资源 + 操作」**（如 `article:write`）若需要，可在角色之上再引入「权限」表；本文先以**角色**为主，预留扩展点即可。
- **组织 / 租户（多租户）**：若未来要按租户隔离角色，需单独设计 `tenant_id`；当前按单租户假设书写。

---

## 2. 概念模型

### 2.1 实体

| 实体 | 说明 |
|------|------|
| **User** | 已有，业务主体。 |
| **Role** | 命名角色，如 `user`、`admin`、`editor`。代码与 JWT 中建议用**稳定英文 code**（如 `admin`），展示名可另字段或走 i18n。 |
| **User ↔ Role** | 多对多：一个用户多个角色，一个角色对应多个用户。 |

### 2.2 关系示意

```text
users          user_roles          roles
─────          ──────────          ─────
user_id ←────  user_id      ───→  uid
email          role_id      ───→  code (unique)
...            (复合唯一)         name (可选)
```

- **关联表**（推荐命名 `user_roles`）：`(user_id, role_id)` 联合主键或唯一约束，避免重复分配。
- **索引**：`user_roles.user_id`、`user_roles.role_id` 便于按用户查角色、按角色反查用户。

### 2.3 预置角色（建议）

| code | 含义 | 说明 |
|------|------|------|
| `user` | 普通用户 | 注册成功后可默认赋予，保证未显式配置时也有基础身份。 |
| `admin` | 管理员 | 运维/后台接口，仅人工或种子数据赋予。 |

预置角色可通过 **数据迁移（seed）** 或 **应用启动幂等脚本** 插入 `roles` 表，避免硬编码仅存在于代码中。

## 3. 与 JWT 的关系

### 3.1 两种常见策略

**策略 A：JWT 内嵌角色列表（推荐起步）**

- 登录成功后，查询用户角色，将 `roles: ["user", "admin"]` 写入 JWT payload（或写入 `realm_access` 风格嵌套，团队统一即可）。
- **优点**：网关或各服务可本地校验签名后直接使用角色，少打用户库。
- **缺点**：改角色后**已签发 token 在过期前仍带旧角色**；若需「立即失效」，需配合黑名单、缩短过期时间或版本号 `token_version` 等机制。

**策略 B：JWT 仅带用户 id，每次请求查库/缓存拉角色**

- **优点**：权限变更实时。
- **缺点**：依赖存储或缓存，延迟与一致性要单独设计。

**建议**：当前 7 天有效期下，采用 **策略 A**，并在文档中明确「角色变更对存量 token 的生效延迟」；若后续强需求再演进到 B 或 hybrid（短 access + 长 refresh 等）。

### 3.2 Payload 字段约定（示例）

- `sub`：用户主键 **`user_id`（UUID 字符串）**。
- `email`：可选，已有。
- `roles`：`string[]`，元素为角色 **code**（与 `roles.code` 一致）。

## 4. API 与鉴权层

### 4.1 注册 / 登录

- **注册**：用户创建成功后，在事务内为其插入 `user_roles`，关联预置角色 `user`（若采用默认角色）。
- **登录**：在签发 JWT 前查询该用户全部角色 code，写入 token。

### 4.2 受保护路由

- FastAPI 依赖项：`get_current_user`（解析 JWT） + `require_roles("admin")` 或 `require_any_role(...)`。
- 未登录：`401`；已登录但角色不足：`403`，`detail` 可与现有项目风格统一（如「权限不足」）。

### 4.3 管理类接口（后续迭代）

- 管理员为用户增删角色：`PATCH /users/{id}/roles` 等，需单独权限与审计，不在「最小可用」内也可先不做，仅种子数据赋 `admin`。

---

## 5. 数据迁移与兼容

1. 新建 `roles`、`user_roles` 表及约束、索引。
2. **数据回填**：为所有已有 `users` 插入 `user` 角色关联（若产品要求「老用户等同普通用户」）。
3. 部署顺序：先迁移表结构与种子角色，再发版应用逻辑（读角色 + 写 JWT），避免空表导致登录失败。

---

## 6. 测试建议

- 单元测试：JWT 中 `roles` 与库一致；依赖项在缺角色时返回 `403`。
- 迁移测试：升级后老用户具备默认角色。

## 8. 参考（概念）

- RBAC：基于角色的访问控制。
- 原则：**角色表规范化存储**；**JWT 携带角色 code 列表**适合当前阶段；**权限矩阵**复杂时再拆 `permissions` 与 `role_permissions`。
